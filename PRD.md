## 1. Problem Statement

Quebec has approximately 2 million residents without a family doctor. When a non-emergency health concern arises, they have three options, all of which are broken:

| Option | Problem |
|---|---|
| **Info-Santé (811)** | Phone-only, synchronous, poorly rated, actively shutting down locations |
| **Emergency Room** | 50%+ of ER visits are non-urgent — patients flood ERs because there's no structured alternative |
| **Google / self-search** | Unstructured, anxiety-amplifying, no clinical grounding, no actionable next step |

There is no structured, asynchronous, accessible way for a patient to submit a health concern and receive a competent, evidence-grounded answer to: **"Should I worry about this, and what do I do next?"**

---

## 2. Product Vision

ContextMD is an asynchronous medical triage navigation platform. Patients submit a structured symptom report — including photos and specific questions — and receive a tiered triage response grounded in NIH public data. For simple cases, Claude generates the self-care response directly. For everything else, a licensed provider reviews the case and authors the response.

ContextMD does not diagnose. It navigates.

---

## 3. Target Users

### Primary: Patients Without a Family Doctor (Quebec)
- Age 18–65
- No access to a regular physician
- Experiencing a non-emergency health concern they don't know how to act on
- Comfortable using a web browser on desktop or mobile
- May have limited health literacy — plain language is essential

### Secondary: Healthcare Providers (Demo Role)
- Nurses, nurse practitioners, or physicians
- Use the provider dashboard to review AI-organized case briefs and author triage responses
- In the hackathon demo: operated manually to demonstrate the workflow

## 4. Core User Flows

### 4.1 Patient Flow

```
App open
  └─ Emergency gate (binary: "Are you in an emergency right now?")
       ├─ YES → Hard redirect: "Call 911 now." No further interaction.
       └─ NO  → 8-step structured intake form
                  1. Body location (region + sub-location)
                  2. Symptom type + description
                  3. Timeline (start date + trend: better/worse/same)
                  4. Severity (pain 0–10 + associated symptoms checkboxes)
                  5. Photo upload — up to 3 images (simulated in hackathon)
                  6. Free text ("describe in your own words")
                  7. Specific questions — up to 3 (e.g. "should I see a dermatologist?")
                  8. Medical history — conditions, medications, allergies
                       └─ Submit
                            ├─ Tier 4 keyword detected → "Go to ER or call 911"
                            └─ No Tier 4 → case created → AI processing begins
                                 └─ Patient sees case status page (polls every 5s)
                                      ├─ Tier 0: response auto-generated by Claude (NIH-grounded)
                                      └─ Tier 1–3: response authored by provider
                                           └─ Patient sees structured response card
```

### 4.2 Provider Flow

```
Provider login (role toggle — no real auth in hackathon)
  └─ Worklist: table of open cases sorted by tier (desc) then submission time (asc)
       └─ Claim case (locks it — no double-claiming)
            └─ Case detail view (two-column)
                 ├─ LEFT: AI clinical brief + patient free text + questions + photos + history
                 └─ RIGHT: Outcome selector + conditional fields + message box
                      └─ Submit response → patient notified → case closed
```

## 5. Triage Tier System

The triage tier is the core of ContextMD's clinical logic. It is computed by Claude using an explicit, non-black-box scoring system defined in the system prompt.

| Tier | Label | Meaning | Who Handles |
|---|---|---|---|
| **4** | Escalate Immediately | Red flag keywords detected at intake | Blocked at form — patient redirected to 911/ER |
| **3** | Urgent | Pain ≥8/10, rapidly worsening, systemic symptoms | Provider must respond within 24–48h |
| **2** | Book Appointment | Concerning features, changing symptoms, needs evaluation | Provider routes to specialist with timeframe |
| **1** | Monitor | Stable, low-severity, single symptom, reassurance needed | Provider provides watch-and-wait guidance |
| **0** | Self-Manageable | Clearly benign, NIH-documented self-care condition | Claude auto-generates response (MedlinePlus-grounded) |

**Tier 4 trigger keywords** (checked at intake submission, client-side, no AI call made):
- Chest pain, difficulty breathing, loss of consciousness, stroke symptoms, severe allergic reaction / anaphylaxis, uncontrolled bleeding, seizure, overdose, suicidal ideation, heart attack

## 6. AI Layer — Claude's Role

Claude performs two functions per case submission (single API call):

### Function 1: Clinical Brief Generation
Converts raw patient input into a structured summary for provider review:
- Chief complaint (one sentence)
- Timeline (when + how it evolved)
- Severity context (pain score + associated symptom burden)
- Red flags present (if any)
- Medication flags (RxNorm + OpenFDA cross-reference)
- Relevant medical history
- Patient's specific questions (prominently listed)
- NIH context (1–2 sentences summarising what retrieved evidence indicates)

This is **organisation and summarisation, not interpretation or diagnosis.**

### Function 2: Complexity Scoring
Assigns a tier (0–4) using explicit, documented criteria in the system prompt. The reasoning is stored and shown to the provider.

### Function 3: Tier 0 Self-Care Response (conditional)
For cases scoring Tier 0, Claude generates a patient-facing self-care response:
- What this likely is (plain language, no diagnostic language)
- What to do at home (grounded in MedlinePlus content)
- Specific symptoms that would warrant escalating to a provider
- Mandatory disclaimer: "This is triage navigation guidance, not a medical diagnosis."

## 7. NIH API Integrations

All APIs are free, public, and require no authentication.

| API | Source | Use |
|---|---|---|
| **MedlinePlus Connect** | NIH / NLM | Patient-readable condition info; powers Tier 0 self-care responses |
| **PubMed E-utilities** | NCBI | Evidence grounding for clinical briefs; red flag pattern matching |
| **RxNorm** | NLM | Medication validation and standardisation from patient-submitted med lists |
| **OpenFDA FAERS** | FDA | Adverse event data for medication-related symptoms |

These are called at triage time (in the `/api/triage` route handler) and their results are passed to Claude in the user message. Claude cites specific retrieved content — it does not fabricate citations.

---

## 8. Provider Response Outcomes

Providers select exactly one outcome per case:

| Outcome | Colour | Conditional Fields |
|---|---|---|
| **Self-Manageable** | Green | None (message box only) |
| **Monitor** | Yellow | Follow-up days + "watch for these symptoms" |
| **Book Appointment** | Orange | Specialist type + recommended timeframe |
| **Urgent** | Red | Mandatory urgency note (what to do and how quickly) |

The provider also writes a plain-language message (no character limit) that directly addresses the patient's specific questions.

## 9. Ethical Constraints (Non-Negotiable)

These constraints are core to the product's validity and are evaluated by judges under the Ethical Alignment criterion (25% of score):

1. **Hard 911 gate.** No patient proceeds to intake if they check any emergency symptom. Full stop.
2. **No diagnosis language.** Claude's system prompt explicitly prohibits diagnostic framing. The product is triage navigation.
3. **Provider-authored clinical responses.** Triage outcomes for Tier 1–3 are written by a human provider, not the AI.
4. **Evidence-grounded AI output.** Tier 0 self-care responses cite specific MedlinePlus content retrieved at runtime. No fabricated sources.
5. **Mandatory disclaimer on every response.** "This is triage navigation guidance, not a medical diagnosis. If your symptoms worsen or you have concerns, contact a healthcare provider."
6. **Transparent tier scoring.** The tier scoring criteria are explicit in the system prompt — not a black box. Providers see the tier and the reasoning.

## 11. Functional Requirements

### Patient-Facing
- **FR-01:** Emergency gate must appear before any intake step. Checking any emergency symptom must show 911 call screen with no path forward into intake.
- **FR-02:** Intake form must persist state to localStorage on every change (auto-save).
- **FR-03:** Patient questions (up to 3) must be stored and displayed to providers prominently — they are the primary thing the patient wants answered.
- **FR-04:** Tier 4 keyword detection must run client-side at submit, before any API call.
- **FR-05:** Case status page must poll every 5 seconds and update the status badge without a full page reload.
- **FR-06:** Response card must display the NIH sources used, with working links.
- **FR-07:** Every patient-facing response must display the triage disclaimer.

### Provider-Facing
- **FR-08:** Worklist must show only non-Tier-0, non-escalated cases in `awaiting_review` or `in_review` status.
- **FR-09:** Claiming a case must be atomic — a second provider cannot claim an already-claimed case (API returns 409).
- **FR-10:** Provider must be able to see the AI clinical brief, tier score, tier reasoning, patient questions, free text, and medical history on the case detail page.
- **FR-11:** Submitting a response must flip case status to `response_ready` and the case must disappear from the active worklist.

### AI / Backend
- **FR-12:** Claude must return valid JSON matching the `TriageAIResult` schema on every call. JSON parse errors must be caught and the case must fall back to `awaiting_review`.
- **FR-13:** NIH API calls must use `AbortSignal.timeout(5000)` — a slow NIH API must not block case creation.
- **FR-14:** Tier 0 cases must have an auto-created `responses` row with outcome `self_manageable` and the Claude-generated message.

---

## 12. Non-Functional Requirements

- **NFR-01 Performance:** Triage AI route (Claude + 4 NIH APIs) must complete within 20 seconds (NIH APIs called in parallel with `Promise.allSettled`).
- **NFR-02 Availability:** Deployed on Vercel — serverless, no infrastructure to manage during the demo.
- **NFR-03 Testing:** ≥70% line coverage on `lib/` and `app/api/`. Safety-critical path (`checkTier4`) must have 100% branch coverage.
- **NFR-04 Type Safety:** No `any` types in `lib/` or `app/api/`. TypeScript strict mode enabled.
- **NFR-05 Accessibility:** All intake form fields must have associated `<label>` elements. Colour is never the sole indicator of status.

## 13. Data Model Summary

### `cases` table
| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | Auto-generated |
| `patient_id` | TEXT | localStorage-generated UUID (no real auth) |
| `status` | TEXT | `processing` → `awaiting_review` → `in_review` → `response_ready` |
| `tier` | INTEGER | 0–4, set by Claude after triage |
| `ai_brief` | JSONB | `ClinicalBrief` shape |
| `ai_tier_reasoning` | TEXT | Claude's explicit tier rationale |
| `claimed_by` | TEXT | Provider ID (prevents double-claim) |
| *(intake fields)* | various | Body location, symptom, timeline, severity, history, photos |

### `responses` table
| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | Auto-generated |
| `case_id` | UUID FK | References `cases.id` |
| `outcome` | TEXT | `self_manageable` \| `monitor` \| `book_appointment` \| `urgent` |
| `message` | TEXT | Patient-facing plain-language response |
| `nih_sources` | JSONB | Array of `{source, title, url, excerpt}` |
| *(conditional)* | various | `followup_days`, `watch_for`, `provider_type`, `timeframe`, `urgency_note` |
